IntuMessage

The universal message envelope that flows through every pipeline stage

Overview

IntuMessage is the foundational data structure of intu. It is a thin, immutable-intent envelope that wraps raw message data together with transport-specific metadata. Every byte that enters a channel through a source connector is immediately wrapped in an IntuMessage, and every pipeline stage — from preprocessor through postprocessor — receives and returns an IntuMessage.

The design follows three principles:

  • Stage-agnostic — IntuMessage does not store which pipeline stage it is in. Stage tracking belongs to the pipeline context (IntuContext).
  • Transport-aware — IntuMessage carries the transport method (source or destination protocol) and protocol-specific metadata so transformers can inspect and modify routing information.
  • Encoding-preserving — The raw body preserves the original source encoding. A sourceCharset field records the original character set so downstream stages can make informed decisions about encoding.
Pipeline as Code The entire “pipeline as code” vision depends on a clear, stable IntuMessage contract. All pipeline transforms and transports operate on IntuMessage exclusively. Future plugin stages will follow the same contract: Plugin(stage, process(IntuMessage)).

Go Struct

IntuMessage is defined in internal/message/message.go. This is the canonical, server-side representation constructed the moment a source connector finishes receiving data.

go
type Message struct {
    ID            string            // UUID, assigned at creation
    CorrelationID string            // Trace ID across systems (defaults to ID)
    ChannelID     string            // Owning channel, set by runtime
    Raw           []byte            // Original message body bytes
    Transport     string            // Source protocol: "http", "kafka", "file", etc.
    ContentType   ContentType       // Declared data format: "json", "hl7v2", "fhir_r4", etc.
    SourceCharset string            // Original charset (e.g. "iso-8859-1"); empty = UTF-8
    HTTP          *HTTPMeta         // HTTP-specific metadata
    File          *FileMeta         // File-specific metadata
    FTP           *FTPMeta          // FTP/SFTP-specific metadata
    Kafka         *KafkaMeta        // Kafka-specific metadata
    TCP           *TCPMeta          // TCP-specific metadata
    SMTP          *SMTPMeta         // SMTP-specific metadata
    DICOM         *DICOMMeta        // DICOM-specific metadata
    Database      *DatabaseMeta     // Database-specific metadata
    Metadata      map[string]any    // Arbitrary key-value metadata
    Timestamp     time.Time         // Time the message was received
}

Transport Metadata

Each protocol has a dedicated metadata struct. Only the struct matching Transport is populated — the rest are nil.

Transport Metadata Struct Key Fields
http HTTPMeta Headers, QueryParams, PathParams, Method, StatusCode
file FileMeta Filename, Directory
ftp / sftp FTPMeta Filename, Directory
kafka KafkaMeta Headers, Topic, Key, Partition, Offset
tcp TCPMeta RemoteAddr
smtp / email SMTPMeta From, To, Subject, CC, BCC
dicom DICOMMeta CallingAE, CalledAE
database DatabaseMeta Query, Params

TypeScript Interface

When IntuMessage crosses the Go ↔ TypeScript boundary, it is serialized as JSON. Every scaffolded project includes src/types/intu.d.ts with globally-available type declarations. Transformers, validators, and filters all receive an IntuMessage as their first argument.

typescript
interface IntuMessage {
  body: unknown;
  transport?: string;
  contentType?: string;

  http?: IntuHTTP;
  file?: IntuFile;
  ftp?: IntuFTP;
  kafka?: IntuKafka;
  tcp?: IntuTCP;
  smtp?: IntuSMTP;
  dicom?: IntuDICOM;
  database?: IntuDatabase;
}
Body Type In the pipeline, body is the parsed payload (e.g. a JSON object if the inbound data type is json). In .intuJSON storage format, body is the raw string (or base64-encoded object for binary).

Lifecycle

IntuMessage is created once and flows through the entire pipeline. Different views of the same message are used at different stages, but the identity (ID, CorrelationID) remains constant within a single processing run.

1. Construction (Source Connector)

When a source connector finishes receiving data, it creates an IntuMessage with message.New(channelID, rawBytes). This assigns a UUID, sets the timestamp, and initializes the metadata map. The connector then stamps Transport and the matching protocol metadata.

go
// HTTP source example
msg := message.New("", body)
msg.Transport = "http"
msg.SourceCharset = encoding.ExtractCharset(r.Header.Get("Content-Type"))
http := msg.EnsureHTTP()
http.Headers = extractHeaders(r)
http.Method = r.Method

2. Channel Assignment

The channel runtime sets msg.ChannelID before any pipeline processing begins. The message is then stored as "received" if storage mode permits.

3. Pipeline Processing (Go → TypeScript → Go)

The pipeline converts the Go Message struct into a map[string]any via buildIntuMessage(), which is then JSON-serialized and sent to the Node.js worker process. TypeScript functions receive it as an IntuMessage object and return a (possibly modified) IntuMessage. The pipeline then parses the result back into a Go Message via parseIntuResult().

This round-trip is designed to be lossless for the fields that transformers can read and modify: body, transport, contentType, and all transport metadata.

4. Destination Pipeline

For each destination, a separate buildDestIntuMessage() constructs a destination-oriented IntuMessage where transport metadata reflects the outbound protocol (from destination config) rather than the inbound source. Destination transformers see the outbound transport context.

5. Storage

At each pipeline milestone (received, transformed, filtered, sent, error, response), the message may be stored via ToIntuJSON(). Storage behavior depends on mode:

Mode Behavior
none No storage at all
status Stores record metadata (ID, stage, status, timestamp) but NOT the message body
full Stores the complete IntuMessage JSON envelope including body

6. Reprocess / Replay

Stored messages can be replayed via the CLI (intu reprocess) or the dashboard. The CLI path uses FromIntuJSON() to reconstruct a full Message from the stored envelope, preserving transport metadata. The message gets a new ID and is re-run through the full pipeline with metadata["reprocessed"] = true and metadata["original_message_id"] set for traceability.

IntuJSON Format

.intuJSON is the canonical serialization format for IntuMessage. It is used for persistent storage, message export, and reprocessing. The format is produced by Message.ToIntuJSON() and consumed by message.FromIntuJSON().

json
{
  "body": "MSH|^~\\&|SENDER|FAC|RECEIVER|FAC|...",
  "transport": "tcp",
  "contentType": "hl7v2",
  "tcp": {
    "remoteAddr": "10.0.1.50:4321"
  }
}

Binary Body Encoding

If the raw body is valid UTF-8, it is stored as a plain JSON string. Otherwise, it is base64-encoded to prevent data corruption:

json
{
  "body": {
    "base64": "SGVsbG8gV29ybGQ=",
    "size": 11
  },
  "transport": "dicom",
  "contentType": "binary"
}

Go ↔ TypeScript Serialization

The pipeline boundary between Go and TypeScript is a JSON bridge over stdin/stdout to Node.js worker processes. IntuMessage is converted between the two runtimes as follows:

Direction Method Description
Go → TS buildIntuMessage() Builds map[string]any from Message. Body is the parsed payload (not raw bytes). JSON-serialized via encoding/json.
TS → Go parseIntuResult() Parses the transformer's return map[string]any back into body + Message fields. Restores contentType, transport, and all transport metadata.
Go → Storage ToIntuJSON() Serializes the raw bytes as body (string or base64). Includes transport and protocol metadata.
Storage → Go FromIntuJSON() Reconstructs Message from stored JSON. Restores raw bytes, transport, and protocol metadata.

IntuContext

Pipeline context is passed alongside IntuMessage as a separate IntuContext object. It carries channel identity, message identity, data type hints, and shared map variables. IntuContext is intentionally separate from IntuMessage to keep the message stage-agnostic.

typescript
interface IntuContext {
  channelId: string;
  correlationId: string;
  messageId: string;
  timestamp: string;           // ISO 8601 / RFC 3339
  inboundDataType?: string;    // from channel data_types.inbound
  outboundDataType?: string;   // from channel data_types.outbound
  destinationName?: string;    // set in destination pipeline stages
  sourceMessage?: IntuMessage; // original source IntuMessage (in dest stages)
  globalMap: IntuMap;
  channelMap: IntuMap;
  responseMap: IntuMap;
  connectorMap?: IntuMap;
}

Structure Audit

This section documents gaps between the current implementation and the product vision for IntuMessage. Each finding includes severity, impact, and a recommended fix.

GAP-1: Two IntuMessage JSON Shapes

Severity: High The pipeline serialization (buildIntuMessage) and storage serialization (ToIntuJSON) produce structurally different JSON for the same message. Pipeline body is parsed data; storage body is raw string/base64. Neither shape includes sourceCharset or the generic metadata map.

Impact: Confusing contract — "IntuMessage" means different things depending on context. Transformers cannot access sourceCharset or custom metadata set by source connectors. Storage envelope is incomplete for faithful reconstruction.

Recommendation: Unify the shapes. Add sourceCharset and metadata to both serialization paths. The body type difference (parsed vs raw) is intentional but should be explicitly documented as "pipeline form" vs "storage form".

GAP-2: SourceCharset Missing from Pipeline IntuMessage

Severity: High msg.SourceCharset is set by source connectors (e.g. HTTP extracts charset from Content-Type) but is not included in the map[string]any passed to TypeScript. The TypeScript IntuMessage interface has no sourceCharset field.

Impact: Transformers processing multi-encoding healthcare data (common with HL7v2 from legacy systems sending ISO-8859-1 or Windows-1252) cannot determine the original encoding. This breaks the vision of IntuMessage as the sole data carrier.

Recommendation: Add sourceCharset to buildIntuMessage(), ToIntuJSON(), FromIntuJSON(), and the TypeScript IntuMessage interface.

GAP-3: Metadata Map Missing from Pipeline IntuMessage

Severity: High Sources set custom metadata (e.g. metadata["filename"] from the file source, metadata["reprocessed"] from reprocessing). This Metadata map is NOT included in buildIntuMessage().

Impact: TypeScript transformers cannot access connector-set metadata. Custom metadata set during preprocessing is invisible to downstream stages. This severely limits the usefulness of the metadata map and blocks the plugin extensibility vision.

Recommendation: Add metadata to both pipeline and storage IntuMessage serialization. Include it in the TypeScript interface as metadata?: Record<string, unknown>.

GAP-4: ToIntuJSON() Does Not Round-Trip Identity Fields

Severity: High ToIntuJSON() does not include id, correlationId, timestamp, channelId, or sourceCharset. FromIntuJSON() calls message.New() which generates fresh UUID and timestamp. A round-trip through ToIntuJSON() → FromIntuJSON() produces a message with different identity.

Impact: Export and re-import of messages is lossy. The product vision requires that "import, export, replay IntuMessage should be smooth and produce 100% predictable outcome". Currently, only the CLI reprocess path compensates by restoring CorrelationID from the MessageRecord wrapper.

Recommendation: Include id, correlationId, timestamp, channelId, sourceCharset, and metadata in ToIntuJSON(). Update FromIntuJSON() to restore these fields instead of generating new ones. This makes .intuJSON a fully self-contained, portable envelope.

GAP-5: Dashboard Reprocess Path Inconsistency

Severity: Medium The CLI reprocess path (cmd/reprocess.go) uses FromIntuJSON() to rebuild messages, preserving transport metadata. The dashboard reprocess path (cmd/serve.go callback) constructs message.New(channelID, rawContent) directly, losing transport metadata.

Impact: Messages replayed via the dashboard may behave differently than those replayed via the CLI, breaking the predictability guarantee.

Recommendation: Unify both reprocess paths to use FromIntuJSON(). Extract the shared logic into a message.Rebuild() or similar function.

GAP-6: No Formal .intuJSON Specification

Severity: Medium There is no JSON schema, file extension convention, or version field for the IntuMessage JSON envelope. The format exists only as Go code in ToIntuJSON() / FromIntuJSON().

Impact: No forward/backward compatibility guarantees. External tools cannot validate .intuJSON files. Plugin authors have no formal specification to code against.

Recommendation: Add a version field (e.g. "version": "1") to the envelope. Create a JSON schema at docs/schema/intumessage.schema.json. Define .intuJSON as the canonical file extension for exported messages.

GAP-7: No Plugin Interface for Custom Pipeline Stages

Severity: Medium (Future) Pipeline stages are hard-coded in pipeline.go. There is no middleware chain or registration mechanism for custom stages between listener and destination.

Impact: The vision of Plugin(stage, process(IntuMessage)) cannot be realized without a plugin host interface. Custom processing (audit logging, enrichment, compliance checks) must currently be embedded in transformers rather than composed as independent stages.

Recommendation: Design a PipelineStage interface in Go:

go
type PipelineStage interface {
    Name() string
    Phase() Phase  // BeforeValidation, AfterTransform, BeforeDestination, etc.
    Process(ctx context.Context, msg *message.Message) (*message.Message, error)
}

Register stages via channel YAML or a plugin config block. Execute them at the declared phase in the pipeline loop. This requires adding an explicit stage field to IntuContext so plugins know where they are.

GAP-8: Context Missing Explicit Stage Field

Severity: Medium buildPipelineCtx() does not include a stage field. The current stage is implicit based on which pipeline function is called.

Impact: Reusable TypeScript modules that behave differently per stage cannot introspect their execution context. Plugin support requires stage awareness.

Recommendation: Add stage?: string to IntuContext. Set it in each pipeline step: "preprocessor", "validator", "source_filter", "transformer", "destination_filter", "destination_transformer", "response_transformer", "postprocessor".

GAP-9: Redis Queue Uses Separate Serialization

Severity: Low redisQueueItem in internal/retry/redis_queue.go mirrors message.Message field-by-field with JSON struct tags. This is a separate serialization path that must be kept in manual sync.

Impact: If fields are added to Message (e.g. sourceCharset, metadata), the Redis queue struct must be updated separately or data is lost during retry.

Recommendation: Replace redisQueueItem with ToIntuJSON() / FromIntuJSON() for Redis serialization, once GAP-4 is resolved and the envelope includes identity fields.

GAP-10: Transformers Docs Incomplete

Severity: Low The Transformers documentation table lists HTTP, File, FTP, Kafka, and TCP transport metadata but omits SMTP, DICOM, and Database.

Recommendation: Add the missing transport metadata entries to the Transformers page IntuMessage table.

Design Principles

When modifying or extending IntuMessage, follow these guiding principles:

  • Single carrier — All pipeline data must flow through IntuMessage. No side-channel state.
  • Serialize once, deserialize once — Minimize the number of serialization formats. Prefer .intuJSON everywhere.
  • Transport metadata is a union, not inheritance — Each protocol gets an optional struct. Only one is populated at a time. No polymorphism.
  • Identity is sacred — ID and CorrelationID must survive serialization round-trips for traceability.
  • Encoding fidelity — Raw bytes are never silently transcoded. The original charset is always recorded.
  • Stage independence — IntuMessage must never assume which pipeline stage it is in. Stage context belongs in IntuContext.
  • Plugin-ready — Every field that plugins might need must be accessible. Prefer inclusion over omission.
